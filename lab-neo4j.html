Setup & Installation

Docker Installation

Start Neo4j using Docker. This sets up a complete Neo4j instance with the web-based Neo4j Browser.

bash 
docker run -d \
  --name neo4j-container \
  -p 7474:7474 \
  -p 7687:7687 \
  -e NEO4J_AUTH=neo4j/password123 \
  neo4j:latest

This command:

  • Runs Neo4j in a detached container
  • Maps port 7474 for the web interface (Neo4j Browser)
  • Maps port 7687 for the Bolt protocol (programmatic access)
  • Sets the default password to "password123"

Access Neo4j Browser

  1. Open your browser and navigate to http://localhost:7474
  2. Login with username neo4j and password password123
  3. You'll be prompted to change the password on first login
  4. Create a new password and remember it for future sessions
ℹ Info:

If port 7474 is already in use, map it to a different port: -p 8080:7474

Theory: Graph Databases

What is a Graph Database?

A graph database is a type of NoSQL database optimized for storing and querying highly connected data. Instead of storing data in tables with rows and columns (relational model), graph databases use a property graph model consisting of:

Nodes

Represent entities (e.g., Product, Customer, Category). Each node can have labels (tags) and properties (key-value pairs).

cypher 
(product:Product {id: "P001", name: "Laptop", price: 999.99})

Relationships

Represent connections between nodes. They have a type (direction) and can also have properties.

cypher 
(product)-[rel:BELONGS_TO {since: "2024-01-15"}]->(category)

Properties

Key-value pairs attached to nodes or relationships. Properties store atomic values (strings, numbers, booleans, dates).

Why Graphs Beat Tables

Aspect Relational (SQL) Graph (Neo4j)
Relationships Foreign keys, JOIN operations Native first-class objects
Performance Slows with many JOINs (O(n)) Constant time traversal (O(1))
Pattern Matching Complex multi-table queries Natural graph patterns
Semantics Limited relationship meaning Rich semantic relationships

Use Cases for Graph Databases

  • Social Networks: Finding friends, mutual connections, community detection.
  • Recommendations: "Customers who bought X also bought Y" or content recommendations.
  • Fraud Detection: Identifying suspicious patterns in transaction networks.
  • Knowledge Graphs: Wikipedia, Google Knowledge Graph, semantic search.
  • Identity & Access: Managing complex permission hierarchies.
  • IoT & Sensors: Modeling device relationships and data flows.

MegaStore Graph Data Model

We'll model the MegaStore e-commerce platform as a graph. Here's the conceptual structure:

Node Types

Label Description Example Properties
Product An item for sale id, name, price, sku, stock
Customer A buyer id, name, email, country, signup_date
Category Product classification id, name, slug
Supplier Product vendor id, name, country, rating

Relationship Types

Type Connects Meaning
PURCHASED Customer → Product Customer bought product (with quantity, date)
BOUGHT_TOGETHER Product → Product Products frequently bought together
SIMILAR_TO Product → Product Products are similar (alternative)
BELONGS_TO Product → Category Product is in category
SUPPLIES Supplier → Product Supplier provides product
REFERRED_BY Customer → Customer Customer referral relationship

Exercise 1: First Steps with Cypher

Learn the basics of Cypher, Neo4j's query language. We'll create nodes, match patterns, and return results.

CREATE: Add Nodes to the Graph

The CREATE statement adds new nodes to the graph.

cypher 
CREATE (p:Product {
  id: "P001",
  name: "Gaming Laptop",
  price: 1299.99,
  stock: 15
})
RETURN p;
Task 1.1:

Create 3 product nodes with different names and prices. Use the code above as a template.

MATCH: Find Nodes

The MATCH statement finds nodes matching a pattern.

cypher 
MATCH (p:Product)
WHERE p.price > 1000
RETURN p.name, p.price
ORDER BY p.price DESC;
Task 1.2:

Write a query to find all products with price between 500 and 1500. Use MATCH and WHERE clauses.

CREATE Relationships

Connect nodes with relationships using the CREATE statement.

cypher 
MATCH (c:Customer {id: "C001"}), (p:Product {id: "P001"})
CREATE (c)-[r:PURCHASED {
  quantity: 2,
  purchase_date: "2024-02-15",
  total_price: 2599.98
}]->(p)
RETURN r;
Task 1.3:

Create a customer node and establish a PURCHASED relationship to one of your products.

Traverse Relationships

Follow relationships to discover connected nodes.

cypher 
MATCH (c:Customer {id: "C001"})-[r:PURCHASED]->(p:Product)
RETURN c.name, r.quantity, p.name, r.purchase_date;

Find all products purchased by a specific customer. Modify the query with a customer ID of your choice.

  email: row.email,
  country: row.country,
  signup_date: row.signup_date,
  loyalty_tier: row.loyalty_tier

Load Purchase Relationships

Load customer purchases from customer_relationships.csv.

cypher 
LOAD CSV WITH HEADERS FROM "file:///customer_relationships.csv" AS row
MATCH (c:Customer {id: row.customer_id})
MATCH (p:Product {id: row.product_id})
CREATE (c)-[r:PURCHASED {
  quantity: toInteger(row.quantity),
  purchase_date: row.purchase_date,
  total_price: toFloat(row.total_price)
}]->(p);

Load Categories

Create category nodes from categories.json.

cypher 
LOAD CSV WITH HEADERS FROM "file:///categories.csv" AS row
CREATE (cat:Category {
  id: row.category_id,
  name: row.category_name,
  slug: row.slug,
  description: row.description
});

Create BELONGS_TO Relationships

Connect products to their categories from product_relationships.csv.

cypher 
LOAD CSV WITH HEADERS FROM "file:///product_relationships.csv" AS row
MATCH (p:Product {id: row.product_id})
MATCH (c:Category {id: row.category_id})
WHERE row.relationship_type = "BELONGS_TO"
CREATE (p)-[r:BELONGS_TO]->(c);

Create Indexes

Create indexes for faster queries on common properties.

cypher 
CREATE INDEX idx_product_id FOR (p:Product) ON (p.id);
CREATE INDEX idx_customer_id FOR (c:Customer) ON (c.id);
CREATE INDEX idx_category_id FOR (cat:Category) ON (cat.id);
CREATE INDEX idx_product_price FOR (p:Product) ON (p.price);

Place CSV files in the Neo4j import/ directory before running LOAD CSV commands.

Exercise 3: Graph Queries

Discover patterns in the MegaStore graph by writing advanced Cypher queries.

Find Related Products

Find products that are similar or frequently bought together.

cypher 
MATCH (p1:Product {id: "P001"})-[r:BOUGHT_TOGETHER|SIMILAR_TO]-(p2:Product)
RETURN p1.name, TYPE(r) AS relationship, p2.name, p2.price
LIMIT 20;
Task 3.1:

Find all products related to a laptop in your database. How many similar or frequently bought-together products are there?

Customer Purchase Paths

Analyze customer purchase behavior and find frequent paths.

cypher 
MATCH (c:Customer {id: "C001"})-[p1:PURCHASED]->(prod1:Product)
MATCH (c)-[p2:PURCHASED]->(prod2:Product)
WHERE p1.purchase_date < p2.purchase_date
RETURN prod1.name, p1.purchase_date, prod2.name, p2.purchase_date
ORDER BY p1.purchase_date;
Task 3.2:

Find a customer's complete purchase history in chronological order.

Category Hierarchies

Explore products within categories and sub-categories.

cypher 
MATCH (c:Category {id: "CAT_ELECTRONICS"})
OPTIONAL MATCH (c)<-[r:BELONGS_TO]-(p:Product)
RETURN c.name, COUNT(p) AS product_count, AVG(p.price) AS avg_price;
Task 3.3:

Calculate statistics for each category: product count, average price, min/max price.

Shortest Paths

Find the shortest relationship path between two products.

cypher 
MATCH (p1:Product {id: "P001"}), (p2:Product {id: "P050"})
MATCH path = shortestPath((p1)-[*]-(p2))
RETURN length(path) AS hops, [node IN nodes(path) | node.name] AS path_nodes;

Exercise 4: Recommendation Engine

Build a product recommendation system using graph analysis and collaborative filtering.

Collaborative Filtering

Find products purchased by similar customers.

cypher 
MATCH (c1:Customer {id: "C001"})-[:PURCHASED]->(p:Product)<-[:PURCHASED]-(c2:Customer)
WHERE c1 <> c2
WITH c2, COUNT(p) AS common_purchases
ORDER BY common_purchases DESC
LIMIT 5
MATCH (c2)-[:PURCHASED]->(recommended:Product)
WHERE NOT (c1)-[:PURCHASED]->(recommended)
RETURN DISTINCT recommended.name, recommended.price;
Task 4.1:

Find 10 product recommendations for a customer based on what similar customers bought.

Co-Purchase Analysis

"Customers who bought X also bought Y" - the classic recommendation.

cypher 
MATCH (p1:Product {id: "P001"})<-[:PURCHASED]-(c:Customer)-[:PURCHASED]->(p2:Product)
WHERE p1 <> p2
WITH p2, COUNT(c) AS purchase_count
ORDER BY purchase_count DESC
RETURN p2.name, p2.price, purchase_count
LIMIT 10;
Task 4.2:

List the top 10 products most frequently bought alongside a gaming laptop.

Category-Based Recommendations

Recommend products from the same category.

cypher 
MATCH (c:Customer {id: "C001"})-[:PURCHASED]->(p:Product)-[:BELONGS_TO]->(cat:Category)
MATCH (cat)<-[:BELONGS_TO]-(rec:Product)
WHERE NOT (c)-[:PURCHASED]->(rec)
WITH rec, COUNT(*) AS category_score
ORDER BY category_score DESC
RETURN rec.name, rec.price, category_score
LIMIT 10;

Build a hybrid recommendation: combine collaborative filtering, co-purchase, and category-based recommendations. Weight them appropriately.

Exercise 5: Knowledge Graph

Model a semantic ontology for the MegaStore domain and build a knowledge graph.

Define Product Ontology

Create concept nodes and relationships that represent domain knowledge.

cypher 
// Create concept hierarchy
CREATE (electronics:Concept {name: "Electronics", scope: "top_level"})
CREATE (computers:Concept {name: "Computers", scope: "category"})
CREATE (laptops:Concept {name: "Laptops", scope: "subcategory"})

// Define hierarchy
CREATE (electronics)-[:HAS_SUBCONCEPT]->(computers)
CREATE (computers)-[:HAS_SUBCONCEPT]->(laptops)

// Define properties
CREATE (gaming:Attribute {name: "Gaming", type: "capability"})
CREATE (laptops)-[:HAS_ATTRIBUTE]->(gaming);

Semantic Relationships

Create semantic relationships beyond simple purchases.

cypher 
// Link products to concepts
MATCH (p:Product {name: "Gaming Laptop"})
MATCH (concept:Concept {name: "Laptops"})
CREATE (p)-[:INSTANCE_OF]->(concept)

// Create property-based relationships
MATCH (p1:Product), (p2:Product)
WHERE p1.price > 1000 AND p2.price > 1000
CREATE (p1)-[:PRICE_COMPARABLE_TO {difference: abs(p1.price - p2.price)}]->(p2);

Inference & Reasoning

Use graph patterns to derive new knowledge.

cypher 
// Infer: If customer bought product of type X, they're interested in X
MATCH (c:Customer)-[:PURCHASED]->(p:Product)-[:INSTANCE_OF]->(concept:Concept)
WITH c, concept, COUNT(p) AS purchase_count
WHERE purchase_count >= 2
MERGE (c)-[r:INTERESTED_IN {score: purchase_count}]->(concept);

Build a knowledge graph for MegaStore with at least 3 levels of concept hierarchy. Link products to concepts and create semantic relationships.

Bonus Tasks

Graph Algorithms

PageRank

Calculate importance of products based on network structure.

cypher 
CALL gds.graph.project('product_graph', 'Product', 'BOUGHT_TOGETHER')
YIELD graphName, nodeCount, relationshipCount

CALL gds.pageRank.stream('product_graph')
YIELD nodeId, score
RETURN gds.util.asNode(nodeId).name AS product_name, score
ORDER BY score DESC
LIMIT 20;

Community Detection

Find groups of related products using Louvain algorithm.

cypher 
CALL gds.graph.project('products', 'Product', 'SIMILAR_TO')
YIELD graphName, nodeCount, relationshipCount

CALL gds.louvain.stream('products')
YIELD nodeId, communityId
RETURN gds.util.asNode(nodeId).name AS product, communityId
ORDER BY communityId;

Python Integration

Access Neo4j from Python using the official driver.

python 
from neo4j import GraphDatabase

driver = GraphDatabase.driver("bolt://localhost:7687", auth=("neo4j", "password123"))

def get_recommendations(customer_id):
    with driver.session() as session:
        result = session.run("""
            MATCH (c:Customer {id: $cid})-[:PURCHASED]->(p:Product)<-[:PURCHASED]-(c2:Customer)
            MATCH (c2)-[:PURCHASED]->(rec:Product)
            WHERE NOT (c)-[:PURCHASED]->(rec)
            RETURN DISTINCT rec.name, rec.price
            LIMIT 10
        """, cid=customer_id)
        return [dict(record) for record in result]

recommendations = get_recommendations("C001")
print(recommendations)

driver.close()

Export & Visualization

Export graph data for visualization in external tools.

cypher 
// Export as JSON for D3.js visualization
MATCH (p:Product)-[r:BOUGHT_TOGETHER]-(p2:Product)
WITH {nodes: [p, p2], links: [{source: p.id, target: p2.id, weight: r.frequency}]} AS graphData
RETURN apoc.convert.toJson(graphData);

Summary & Troubleshooting

Key Concepts

  • Nodes: Entities with labels and properties
  • Relationships: Typed connections between nodes with properties
  • Cypher: Intuitive query language for pattern matching
  • Performance: O(1) relationship traversal, perfect for connected data
  • Semantics: Rich relationship types enable knowledge representation

Troubleshooting

Port Already in Use:

If port 7474 is busy, use docker run -p 8080:7474 ... and access at localhost:8080

LOAD CSV Not Finding Files:

Ensure CSV files are in /var/lib/neo4j/import/ inside the container. Use docker cp file.csv neo4j-container:/var/lib/neo4j/import/

Out of Memory:

Increase Docker memory: docker run -e NEO4J_dbms_memory_heap_max__size=4g ...

Query Timeout:

Add LIMIT clause or use indexes. Check execution plan with EXPLAIN/PROFILE
